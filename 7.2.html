CHAPTER 7.2: DESIGN

What is Design Stage?

Once the analysis has taken place and the systems analyst has some idea of the scale of the problem and what needs to be done, the next step is to design the key parts of the recommended system

Tasks to be done in the Design Stage

  • Design the data capture forms/input forms
  • Design the screen layouts
  • Design outputs in the form of screen displays and printed reports
  • Produce systems flowcharts and/or pseudo code
  • Select/design validation rules that need to be used
  • Design/agree on file structures
  • Produce any algorithms or program flowcharts
  • Design a testing strategy/plan

Data Capture Forms

  • Allows the data to be input into the system
  • Either paper-based or electronic-based depending on the application
  • Paper-based forms need to:
    • Have a heading to make the purpose of the form clear
    • Make it clear to the person filling in the form where they must place their answers
    • Make use of text boxes, which will limit the amount of information collected
    • Make use of character boxes for data such as surnames, telephone numbers, and so on (each box allows one character only)
    • Make use of printed text boxes to allow for easy input of items such as date of birth
    • Make use of tick boxes to make choices easier (such as sex - male or female)
    • Make sure there is sufficient space to write answers
    • Use clear fonts and clear text colours to ensure the form is easy to read
  • A computer-based data capture form is slightly different and has the following features:
    • Use of text boxes to capture key data clearly
    • Use of on-screen help when completing the form
    • Use of drop-down/combo boxes where there are limited choices
    • Use of radio buttons and tick boxes, requiring a single click of a mouse to select
    • Automatic validation of data as it is entered
    • Control buttons (such as next form, clear entry, save, etc)
    • Double-entry boxes (with verification rules) to check the correctness of key data (for example, when keying in an email address)

Screen displays and printed reports

  • The output from any system needs careful consideration since this is part of any user interface and is also the result of some form of processing
  • Screen outputs should be designed to make sure the size of all the output fields is correct so that:
    • Any instructions/descriptions are clear
    • The full screen is utilised (avoiding large areas of ‘nothing)
    • Colours and fonts (size and type)) make the output clear
  • Reports should:
    • Clearly show all the fields that were included in the search criteria

System flowcharts

  • Used to show how data flows through a system and also how decisions are made
  • Make use of special symbols that represent input/output, processing, decisions, and data storage
  • System analysts use these charts to give an overall view of the proposed system
    • Do not form the basis of a flowchart from which software can be written but they do show the processes are carried out and where various hardware devices are used in the system

Verification

  • A way of preventing errors when data is copied from one medium to another (for example, from paper to disk/CD). - whether the data matches the form
  • Two common ways that verification checks are carried out:
    • Double-entry: data is entered twice, using two people, and is them compared
    • Visual check: checking for errors by the person who is entering the data; compare the entered data with the original document (proof-reading)

Validation

  • A process where data is checked to see if it satisfies certain criteria when input into a computer, for example, to see if the data falls within accepted boundaries
  • Types of validation checks:
    Validation Check Description
    Range check

    E.g. A person’s age should be in the range > 0 but < 150

    		Checks whether data is within given/acceptable values
    Look-up Check Checks whether the data entered exists and is stored in a table of data
    Length check

    E.g. If a field needs six digits then inputting a five- or seven-digit number, for example, should cause an error message

    		 Checks if the input data contains the required number of characters
    Character/type check Checks that the input data doesn’t contain invalid characters
    Format/picture check

    E.g. Date should be in the form dd/mm/yyyy

    		 Checks that data is in a specific format
    Presence Check

    E.g. In an electronic form, a person’s telephone number may be a required field and if no data is present this should give rise to an error message

    		 Checks if data is actually present and hasn’t been missed out
    Consistency Check

    E.g. If ‘Mr’ has been typed into a field called title then the gender field must contain either ‘M’ or ‘Male'

    		 Checks if fields correspond (tie-up) with each other
    Check Digit

    E.g. Check digits can identify three types of error:

    • If two digits have been inverted during input
    • An incorrect digit entered twice
    • A digit missed out altogether

    		 This is an extra digit added to a number which is calculated from the digits
    • Can identify three types of error:
      • If two digits have been transposed during input
      • An incorrect digit entered twice
      • A digit missed out altogether

File structures

  • Designing and agreeing file structures is an important part of the design stage
  • Used in the files need to take the following into account:
    • Field Length
    • Field Name (suitable names should be chosen)
    • Data Type
  • The data dictionary can be used to show suitable field names

      Manufacture_year 4 Numeric Range Check
      Product_name 20 Alphanumeric None
      Location_of_stock 4 Numeric Character Check
      Colour 2 Alphanumeric Look-up Check

    Design and testing strategy/plan

    • When producing the software for the new system, it is very important to test it thoroughly in order to:
      • Make sure it meets the agreed client requirements
      • Remove any bugs/errors from the system
      • Make sure it produces the required output for data where the correct output is already known
      • Check that the software doesn’t crash under certain conditions
