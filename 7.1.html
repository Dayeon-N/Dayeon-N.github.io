CHAPTER 7.1: ANALYSIS

What is Analysis Stage?

Analysis: stage of collecting information about the present system and identifying problems

Steps in the Analysis Stage

  • Fact-finding/collecting data from the current system
  • Description of the current system - establishing the inputs, outputs, and processing being done
  • Identification of the problems with the current system
  • Agreement of the objectives with the clients
  • Identify and agree with customer requirements
  • Interpreting the customer’s requirements
  • Producing a cost-benefit analysis
  • Identifying suitable hardware and software
  • Producing a data flow diagram

Fact-finding

A variety of fact-finding methods are used to gather information and these are:

  • Questionnaires
  • Interviews
  • Observations
  • Examining documents (looking at existing paperwork)

Observations

  • Involves watching personnel using the existing system to find out exactly how it works
Pros Cons
  • Obtains reliable data
  • Possible to see exactly what is being done
  • Relatively inexpensive
  • People are generally uncomfortable being watched
  • If workers perform tasks that violate standard procedures, they may not do this while being watched

Interviews

  • Involves a one-to-one question-and-answer session between the analyst and the employee/customer
Pros Cons
  • Gives opportunity to motivate the interviewee into giving open and honest answers to the analyst’s questions
  • Allows the analyst to probe for more feedback from the interviewee - easy to extend a question
  • Possible to modify questions
  • Able to probe deeply into one specific aspect of the existing system
  • Time-consuming
  • Expensive, due to the use of the analyst’s time
  • The interviewee cannot remain anonymous

Looking at existing paperwork

  • Allows the analyst to see how the paper files are kept, look at operating instructions and training manuals, check the accounts, etc
Pros Cons
  • Allows information to be obtained which was not possible by any of the other methods
  • Analysts can see for themselves how the paper system operates
  • Allows the analyst to get some idea of the scale of the problem, memory size requirements, type of input/output devices needed
  • Can be very time-consuming
  • Can be relatively expensive because of the analyst’s time

Questionnaires

  • Involves sending out questionnaires to the workforce and/or to customers to find out views of the existing system and find out how some key tasks are carried out
Pros Cons
  • Can be answered quite quickly
  • Relatively inexpensive
  • Can remain anonymous
  • Allows quick analysis of the data
  • The number of people who returned questionnaires is low
  • Inflexible questions since they have to be generic
  • No immediate way to clarify a vague or incomplete answer to a question

Data flow diagram (DFD)

  • Help the analyst by showing the data flow, input and output requirements, processing, and the types of data storage needed
  • It is used to describe:
    • The need to identify inputs, outputs, and processing of the current system
    • The need to identify problems with the current system
    • The need to identify the user and information requirements for the new system
    • System specifications:
      • Identify and justify suitable hardware for the new system
      • Identify and justify new software for the new system
