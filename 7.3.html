CHAPTER 7.3: DEVELOPMENT & TESTING

Development

  • If the system contains files
    • File structure needs to be finalised at this stage
    • Once the file structure has been determined, it is then created and tested fully to make sure it is robust when the system actually goes live

  • Techniques need to be adopted to make sure the data populating the files/database is at least of the right type and that it conforms to certain rules
    • Validation and verification are used to ensure this
      • Need to be fully tested to ensure they do trap unwanted data, but also to make sure that any data transferred from a paper-based system to an electronic system has been done accurately

User Interface

  • How the types of hardware are used to interface with the final system now needs to be identified
    • How the screens and any other input devices will be used to collect the data and the way the output will be presented
  • If specialist hardware is needed, how these devices will be used with the system when it is implemented needs to be finalised
    • Followed by thorough testing to ensure that the user screens are user-friendly and that the correct output is associated with the inputs to the system

Testing Strategies

  • Software is often developed in modular form
    • Allows the software to be broken down into smaller parts, known as modules
    • Each part is developed separately by a programmer and is tested to see if it functions correctly
    • Any problem resulting from the testing requires the module to be modified and then tested again
  • After the completion of the development of the module
    • The whole system needs to be tested even though each module may work satisfactorily, when they are all put together there may be data clashes or incompatibility, memory issues, etc
  • All of this may lead to a need to improve the input and output methods, file/database structures, validation, and verification methods, etc and then test everything fully again
    • Very time-consuming process
    • Has to be as perfect as possible

Different types of Data used in Testing

Normal

  • Data that is acceptable/reasonable
  • Has an expected outcome

Extreme

  • Data at the limits of acceptability

Abnormal

  • Data outside the limits of acceptability, or the wrong type of data
  • Should be rejected or cause an error message

Live

  • Once a system has been fully tested, it is then tested with live data, live data is entered into the new system and the results compared with those produced from the existing system
